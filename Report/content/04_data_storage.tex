\section{Data Storage}

\subsection{Data Life-cycle and Storage Architecture}
The system distinguishes between two primary forms of data: \textit{Transient Input Data} and \textit{Structural State Data}. This separation ensures that the raw input remains immutable while the visualization undergoes complex transformations.

\subsubsection{Transient Input Storage}
Before a data structure is constructed, raw values are stored in centralized buffers managed by the \texttt{DataManager}. 
\begin{itemize}
    \item \textbf{Schema Consistency}: Whether the data originates from a \texttt{.txt} file, a random generator, or manual console input, it is normalized into standard containers (\texttt{std::vector}).
    \item \textbf{Buffer Management}: The system follows a "Clear-on-Load" strategy. Each new ingestion command triggers a memory reset of the transient buffers to prevent data leakage between unrelated visualization sessions.
\end{itemize}

\subsubsection{Persistent Structural Representation}
Once the "Build" command is issued, the Model layer (e.g., \texttt{AVLTree}, \texttt{Graph}) takes ownership of the transient data and converts it into a persistent representation:
\begin{itemize}
    \item \textbf{Node-Link Persistence}: Trees and Graphs are stored as pointer-based structures or adjacency lists, where each node maintains its own state (value, color, position).
    \item \textbf{Historical State Buffer}: As the algorithm progresses, the system does not overwrite the current state but stores "Snapshots" (Payloads) in the \texttt{Timeline}. This allows the storage to represent the data not just in its current form, but across its entire execution history.
\end{itemize}
